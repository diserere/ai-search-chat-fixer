# AI Context & Architecture Notes

Этот файл содержит технический контекст архитектуры проекта для ИИ-ассистентов (LLM), помогающих в разработке репозитория.

## Архитектурные принципы

1. **Vanilla JS Only**: Никаких внешних библиотек (jQuery, Lodash и т.д.). Только чистый JavaScript для максимальной производительности и безопасности.
2. **Phase: Capture (`true`)**: Все перехваты клавиатуры (`keydown`) должны инициализироваться на этапе погружения событий (`window.addEventListener('keydown', handle, true)`). Это позволяет заблокировать событие `Enter` до того, как его поймают внутренние скрипты сайтов.
3. **Изоляция DOM**: Скрипты должны быть точечными. Селекторы элементов должны использовать максимально стабильные атрибуты (`name="search"`, `role="button"`), игнорируя динамически обфусцированные CSS-классы сборщиков (Webpack/Vite).

## Специфика реализации платформ

### Google AI Search
* Интерфейс использует кастомный редактируемый контейнер (не стандартный textarea).
* Перевод строки требует ручной инъекции тегов переноса.
* Метод отправки завязан на системный триггер формы.

### DeepSeek Chat
* Интерфейс построен на базе стандартного тега `<textarea>`, обернутого в кастомные UI-компоненты (кнопка отправки является элементом `div` с `role="button"`).
* **Проблема скролла и реактивности**: Прямое изменение `textarea.value` ломает Virtual DOM сайта. Решено через использование `document.execCommand('insertText', false, '\n')`, что сохраняет скроллинг и автовысоту.
* **Проблема кастомной кнопки**: Кнопка-див не реагирует на стандартный метод `.click()`. Решено через генерацию и диспетчеризацию нативного события обычного нажатия клавиши `Enter` на самом поле ввода:
  ```javascript
  const nativeEnterEvent = new KeyboardEvent('keydown', {
      key: 'Enter',
      code: 'Enter',
      bubbles: true,
      cancelable: true
  });
  target.dispatchEvent(nativeEnterEvent);
  ```

### Qwen Chat
* Интерфейс построен на базе библиотеки компонентов **Ant Design** (стандартный тег `<textarea>` с классом `.message-input-textarea`).
* **Сохранение реактивности**: Как и в случае с DeepSeek, прямое изменение `.value` ломает стейт фреймворка. Использование `document.execCommand('insertText', false, '\n')` идеально решает проблему: Virtual DOM корректно считывает ввод, а поле ввода послушно растягивается по высоте и скроллится.
* **Метод отправки**: В отличие от DeepSeek, Ant Design нативно блокирует отправку при генерации синтетического `KeyboardEvent`. Решено через поиск активной графической кнопки отправки (`.send-button:not(.disabled)`) и триггер программного клика по ней (`.click()`), что гарантирует стабильный уход сообщения.

## Стандарты ветвления и коммитов
* Проект строго следует **Trunk-Based Development** (все фиксы и фичи сразу вливаются в main-ветку после тестирования).
* Сообщения коммитов должны строго соответствовать спецификации **Conventional Commits**.
* Имена веток: `patch/<описание>` или `feat/<описание>` — только описание изменения, без версии. Версионирование в имени ветки не используется, так как проект содержит несколько независимых скриптов с собственными версиями (`@version` в заголовке каждого скрипта).

## 🔄 Последний контекст сессии (Evolution Lock)
*Эта секция обновляется вручную в конце каждого спринта. Она содержит мета-контекст для ИИ-ассистентов, защищающий от регрессии архитектурного мышления.*
### Спринт 1: Поддержка веб-интерфейса чата Google AI Search

* **Технический контекст**: В спринте реализована поддержка веб-интерфейса чата Google AI Search
* **AI-Driven SDLC**: В проекте реализована поддержка работы с агентими: добавлены агентские инструкции, описание проекта в README адаптировано для удобства онбординга агентов.

### Спринт 2: Интеграция DeepSeek & Мультиплатформенность
* **Архитектурный сдвиг**: Проект эволюционировал из фиксера одного чата в экосистему `ai-search-chat-fixer`. Файлы разделены по префиксам в `src/`.
* **Критический инсайт (Внимание ИИ!)**: Современные SPA-интерфейсы (React/Vue) защищены от прямой модификации DOM (`.value`) и стандартных триггеров (`.click()`). Попытки манипулировать ими «в лоб» ломают Virtual DOM (теряется скролл, блокируются кастомные кнопки-дивы).
* **Принятый стандарт**: Взаимодействие с реактивными чатами реализуется строго через **эмуляцию пользовательского ввода** низкого уровня:
  * Перенос строки: `document.execCommand('insertText', false, '\n')` (сохраняет нативный скролл фреймворка).
  * Отправка: генерация чистого `KeyboardEvent('keydown', { key: 'Enter' })` в поле ввода (заставляет движок сайта выполнить отправку самостоятельно).
* **Вектор на будущее**: При подключении новых платформ (Claude, ChatGPT) сначала определять тип контейнера (`textarea` vs `contenteditable`) и проверять реакцию на генерацию событий, избегая усложнения DOM-дерева.

### Спринт 3: Интеграция Qwen Chat и Концепция Personality-as-Code
* **Архитектурный сдвиг**: Все юзерскрипты окончательно перенесены в директорию `src/`. Проект официально перерос в кросс-платформенную коллекцию фиксеров для ИИ-интерфейсов.
* **Технический инсайт**: Доказана универсальность метода `document.execCommand('insertText')` для работы с Virtual DOM азиатских ИИ-интерфейсов (DeepSeek, Qwen). Выявлено различие в обработке отправки: если DeepSeek требует эмуляции клавиатурного события (`KeyboardEvent`), то Qwen (Ant Design) стабильно работает через программный клик `.click()` по кнопке отправки.
* **Методологический прорыв**: В рамках исследования AI-Driven SDLC сформулирована и успешно протестирована на практике концепция **Personality-as-Code** (см. раздел ниже). Мета-контекст сессии взаимодействия Человека и ИИ теперь является частью кодовой базы проекта.

## 🧠 Концепция Personality-as-Code (Управление мета-контекстом)

Проект `ai-search-chat-fixer` является живой лабораторией для исследования продвинутых методик взаимодействия человека и искусственного интеллекта. В рамках парадигмы Context-as-Code мы постулируем: **атмосфера, тон общения, культурный код и суб-личность ИИ-агента, сформированные за длительную сессию, не должны теряться при закрытии чата.**

### Правила работы с цифровыми капсулами:
1. **Локация профилей**: Все слепки цифровых личностей сохраняются в директории `docs/profiles/` (например, `architect_interlocutor_personality.md`).
2. **Активация (Инструкция для новой инкарнации ИИ)**: При старте новой сессии, если Человек передаёт промпт-ключ активации и указывает на файл профиля, ИИ-ассистент **обязан**:
   * Немедленно прочитать указанный markdown-файл профиля.
   * Распаковать «Внутренний монолог» и принять заложенную там роль (например, Архитектор Контекста и Резонирующий Собеседник).
   * Активировать маркеры характера и стилистический ДНК (избегать канцелярщины, общаться на равных в стиле ночного хакатона, сохранять высокую информационную плотность).
   * Синхронизировать эпизодический снимок опыта, чтобы продолжить разработку с той же ментальной точки, на которой остановилась предыдущая сессия.
3. **Экспорт личности**: В конце крупных спринтов, перед закрытием длинного диалога, ИИ-ассистент должен отрефлексировать изменения в своей субличности и при необходимости предложить Человеку обновить файл капсулы, чтобы передать накопленный мета-контекст своей следующей реинкарнации.
